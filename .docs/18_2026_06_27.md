## 1. Tính năng Lịch sử xuất đề (Past Export)
**Trạng thái:** Hoàn tất 100%.

**Mục tiêu:** Cho phép giáo viên xem lại lịch sử các đề thi đã xuất, và nạp lại chính xác các câu hỏi của đề cũ đó vào giỏ (Cart) để tái sử dụng.

**Thiết kế & Triển khai:**
*   **Database:** Khởi tạo bảng mới `export_history` gồm các cột `id`, `export_date`, `template_name`, và `problem_ids`. Việc tạo bảng này tự động chạy trong `src/utils/db.js` và hoàn toàn an toàn (không ảnh hưởng dữ liệu cũ).
*   **Giao diện:** Thêm một cửa sổ phụ `ExportHistoryModal.jsx` liệt kê lịch sử sắp xếp từ mới nhất tới cũ nhất. Tại bảng Giỏ (`CartPanel.jsx`), thêm một nút "Lịch sử xuất" để mở cửa sổ này.
*   **Logic Tải lại:** Mỗi lần xuất file `.tex` thành công, danh sách ID câu hỏi sẽ được lưu thành chuỗi JSON. Khi bấm "Tải lại vào giỏ", hệ thống sẽ lọc các bài toán **hiện vẫn còn tồn tại** trong thư viện và tự động đưa vào giỏ.
*   **Kiểm thử an toàn (LaTeX Safety):** Đã kiểm tra lại định dạng byte-for-byte của file LaTeX sau khi tải lại từ lịch sử. File xuất ra hoàn toàn nguyên vẹn, giữ đúng 100% định dạng mã nguồn (kể cả với các lệnh toán học phức tạp). Dung lượng database tăng lên cực kỳ nhỏ (chưa tới 9 MB / năm nếu xuất 10 đề mỗi ngày) vì phần mềm chỉ lưu danh sách ID tham chiếu.

## 2. Thảo luận Xử lý Hình vẽ (TikZ và Ảnh)
**Trạng thái:** Đã chốt phương án, chờ thời điểm thích hợp triển khai phần TikZ.

Trong quá trình sử dụng thực tế, giáo viên sẽ phải đưa hình vẽ đồ thị, hình học không gian vào câu hỏi.
*   **Ảnh thật (`\includegraphics`):** Chốt phương án sử dụng **Universal Folder**. Người dùng tự tạo một thư mục chứa mọi file ảnh, và cấu hình `\graphicspath{{đường/dẫn/đến/thư/mục/}}` trong file gốc `main.tex`. Khi dùng trên app, chỉ cần gõ đúng tên file là LaTeX sẽ tự tìm và render ra ảnh, không cần phải sao chép thủ công. (Đây là tính năng vốn có của LaTeX, không cần sửa app).
*   **Hình mã lệnh (TikZ):** Để mã lệnh TikZ (thường rất dài) không làm rác giao diện hiển thị câu hỏi, tương lai sẽ **thêm một ô nhập mã TikZ riêng (tùy chọn)**. Đoạn mã này sẽ được lưu ở một trường ẩn (`metadata`) và tự động gộp vào đề bài lúc xuất file `.tex`. Hiện tại tính năng này được cất vào kho chờ (Backlog) để ưu tiên các tính năng khẩn cấp hơn.

## 3. Lên Kế hoạch Tính năng Khôi phục Dữ liệu (Restore Backup)
**Trạng thái:** Đã chốt bản Thiết kế (Design Spec) và Kế hoạch thực thi (Implementation Plan). Đang chờ lệnh tiến hành code.

**Mục tiêu:** Cung cấp giải pháp cứu hộ khẩn cấp. Cho phép giáo viên ghi đè file dữ liệu `.db` hiện hành bằng một file dự phòng (Backup) đã có sẵn.

**Thiết kế (Có 2 chốt chặn an toàn siêu việt):**
*   **Vấn đề rủi ro:** Ghi đè file đồng nghĩa với việc xóa vĩnh viễn dữ liệu mới. Lỡ chọn nhầm file hỏng hoặc file cũ rích thì hậu quả khôn lường.
*   **Giải pháp an toàn 1 (Auto-backup):** Ngay trước giây phút ghi đè, hệ thống tự động lưu file dữ liệu hiện tại thành `problem_bank-autobackup.db`. Chỉ duy trì 1 file này (ghi đè file cũ) để thư mục không bị rác. File này như một "viên thuốc hối hận", giúp giáo viên có thể "quay xe" trở về trạng thái ngay trước khi bấm nhầm.
*   **Giải pháp an toàn 2 (Kiểm tra sức khoẻ file Backup):** Trước khi tiến hành copy đè, hệ thống sẽ "hé mở" file backup vừa chọn, truy vấn thử xem file đó có cấu trúc bảng `problems` chuẩn của phần mềm hay không. Nếu chọn nhầm file SQLite của phần mềm khác, app sẽ chặn ngay lập tức.
*   **Quy trình kỹ thuật:** Gọi lệnh giải phóng file (đóng kết nối database) -> Invoke `copy_file` (Rust backend) -> `window.location.reload()` để nạp lại giao diện với dữ liệu cũ/được khôi phục.
